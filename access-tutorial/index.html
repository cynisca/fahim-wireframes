Module 1

The Big Picture

Microsoft Access is a desktop relational database that comes with Microsoft 365. Think of it as Excel's more structured cousin—instead of one giant spreadsheet, your data lives in separate, linked tables with strict rules.

Why do MBAs learn Access?

It teaches you to think in terms of structured data: how businesses store customer records, orders, inventory, and financials so they can query and report on them. The same concepts apply to SQL Server, Oracle, Salesforce—anywhere data lives in a real company.

The four objects in Access

Tables hold the raw data. Queries filter, sort, and calculate. Forms give users a friendly way to enter data. Reports format data for printing or PDFs.

💡 Mental model
Tables = spreadsheet tabs  |  Queries = Excel filters on steroids  |  Forms = data-entry screens  |  Reports = formatted printouts

Tables & Data Types

A table is a grid of rows (records) and columns (fields). Every table needs a Primary Key—a field that uniquely identifies each row (like a student ID or order number).

Field properties to know

Field Size — limits length of text/number. Required — prevents blanks. Default Value — auto-fills a starting value. Validation Rule — enforces constraints (e.g., >0 means only positive numbers). Input Mask — formats input (e.g., phone number pattern).

Module 3

Relationships

The whole point of a relational database is that tables are linked. Instead of repeating a customer's name on every order row, you store the customer once and reference their ID.

The three relationship types

TypeMeaningExample
One-to-Many (1:∞)One record in Table A matches many in Table BOne customer → many orders
One-to-One (1:1)One record matches exactly one in another tableEmployee → employee details
Many-to-Many (∞:∞)Requires a junction table in betweenStudents ↔ Classes (via Enrollments)

How to create a relationship

Go to Database Tools → Relationships. Drag the primary key from one table onto the matching foreign key in another. Check Enforce Referential Integrity — this prevents orphan records (e.g., an order pointing to a customer that doesn't exist).

Module 4

Queries

Queries are the most powerful part of Access. They let you filter, sort, calculate, and combine data from multiple tables.

Select Query (the most common)

Go to Create → Query Design. Add tables, drag fields into the grid, set criteria and sort order. This creates a Select Query—it reads data without changing it.

Criteria syntax cheat sheet

What you wantCriteria
Exact match"New York"
Not equalNot "New York"
Greater than $100>100
Between two datesBetween #1/1/2026# And #3/31/2026#
Starts with "S"Like "S*"
Is blank / not blankIs Null / Is Not Null
Multiple values"NY" Or "CA" Or "TX"

Calculated fields

You can create new columns in the query grid. Just type an expression in a blank column:

Tax: [TotalAmount] * 0.08
GrandTotal: [TotalAmount] + [TotalAmount] * 0.08
FullName: [FirstName] & " " & [LastName]

Totals queries (aggregation)

Click the Σ Totals button in query design. This adds a "Total" row where you pick: Group By, Sum, Avg, Count, Min, Max.

Example: "Total spend per customer" → Group By CustomerID, Sum of TotalAmount.

Other query types

Action queries modify data: Update (change values), Append (insert rows), Delete (remove rows), Make-Table (create a new table from results). Use with caution—they change your data permanently.

Parameter queries prompt the user for input at runtime. Put brackets around a prompt in the criteria: [Enter city name:]

Module 5

Forms

Forms provide a user-friendly interface for data entry instead of editing tables directly. In a real business, end users would never touch the raw tables—they'd use forms.

Quick way to create a form

Select a table in the navigation pane, then click Create → Form. Access auto-generates a layout. Switch to Layout View to rearrange fields, resize, add logos, and change colors.

Key form concepts

Bound vs. Unbound controls: A bound text box is linked to a table field (edits save automatically). An unbound control is standalone (like a label or calculated field).

Subforms: Embed a related table inside a form. Example: a customer form with an orders subform showing all their orders at the bottom.

Combo box: A dropdown that lets users pick from a list (e.g., select a customer from a dropdown instead of typing the ID manually). Use the wizard to set it up.

Module 6

Reports

Reports format data for printing or PDF export. They're read-only—users can't edit data through a report.

Creating a report

Select a query (or table) → Create → Report. Access generates a report. Use Layout View to adjust column widths and add headers/footers.

Grouping & sorting

In Design or Layout View, click Group & Sort. You can group by a field (e.g., group orders by customer) and add subtotals at each group break. The report footer can show grand totals.

Calculated controls in reports

=Sum([TotalAmount])        ← grand total in report footer
=Count(*)                  ← number of records
=Now()                     ← current date/time
="Page " & [Page] & " of " & [Pages]  ← page numbers

Module 7

SQL Basics

Every query in Access has a SQL View behind it. Understanding basic SQL makes you dangerous in any data role.

⚡ Pro tip
In any query, click View → SQL View to see (and edit) the SQL. This is the fastest way to learn SQL—build queries in design view, then read the generated SQL.

Core SQL statements

-- Select specific columns with a filter
SELECT FirstName, LastName, City
FROM tblCustomers
WHERE City = "New York"
ORDER BY LastName;

-- Join two tables
SELECT c.FirstName, c.LastName, o.OrderDate, o.TotalAmount
FROM tblCustomers AS c
INNER JOIN tblOrders AS o
ON c.CustomerID = o.CustomerID
WHERE o.TotalAmount > 50;

-- Aggregation with GROUP BY
SELECT c.LastName, SUM(o.TotalAmount) AS TotalSpend
FROM tblCustomers AS c
INNER JOIN tblOrders AS o
ON c.CustomerID = o.CustomerID
GROUP BY c.LastName
HAVING SUM(o.TotalAmount) > 100;

Mini Project: Coffee Shop Database

Put it all together. Build a small database for a fictional coffee shop called Bean & Brew.

Step 1 — Create these tables

tblProducts: ProductID (AutoNumber, PK), ProductName (Short Text), Category (Short Text: "Coffee", "Tea", "Pastry"), Price (Currency).

tblEmployees: EmployeeID (AutoNumber, PK), FirstName, LastName, HireDate (Date/Time).

tblSales: SaleID (AutoNumber, PK), EmployeeID (Number, FK), ProductID (Number, FK), SaleDate (Date/Time), Quantity (Number).

Add 6 products, 3 employees, and 15 sales records.

Step 2 — Set up relationships

tblEmployees 1:∞ tblSales (on EmployeeID)
tblProducts 1:∞ tblSales (on ProductID)
Enforce referential integrity on both.

Step 3 — Build these queries

qryRevenueByProduct — Calculate Revenue: [Price] * [Quantity], then Group By ProductName, Sum of Revenue. Sort descending.

qrySalesByEmployee — Count of sales and Sum of revenue per employee.

qryCoffeeSalesOnly — All sales where Category = "Coffee".

Step 4 — Create a form and report

frmNewSale — A form for entering new sales. Use combo boxes for EmployeeID (showing names) and ProductID (showing product names).

rptRevenueByProduct — A report based on qryRevenueByProduct. Group by Category, with subtotals and a grand total.
